# Спецификация: Генерация кодов для входа через веб-форму

**Дата:** 2026-04-27  
**Статус:** На утверждение

## Контекст

В настоящий момент Telegram-бот может генерировать коды для входа на веб-сайт, но только если пользователь уже зарегистрирован в системе. Нужно добавить функционал, при котором:

1. На веб-сайте есть кнопка/ссылка "Войти через Telegram"
2. Ссылка содержит фиксированный приглашающий токен
3. Бот, получив этот токен, может:
   - **Если пользователь уже в системе** → сгенерировать и выдать код
   - **Если пользователя нет** → зарегистрировать его и выдать код

## Требования

### Функциональные

1. **Единая пригласительная ссылка**: `https://t.me/bot_username?start=INVITE_TOKEN`
   - `INVITE_TOKEN` — фиксированная строка для всех пользователей
   - Хранится в переменных окружения (`INVITE_TOKEN`)

2. **Обработка в боте** (`/start INVITE_TOKEN`):
   - Валидирует токен
   - Проверяет наличие пользователя в системе
   - Генерирует 8-символьный код (A-Z, 0-9)
   - Отправляет код пользователю текстом: "Ваш код для входа: `ABC12345`"

3. **Регистрация новых пользователей**:
   - Передает данные на бэкенд (`POST /api/v1/auth/register-from-invite`)
   - Бэкенд валидирует токен, создает пользователя
   - Возвращает сгенерированный код

4. **Хранение кодов**:
   - Сохраняются в таблице `login_codes` (веб-БД)
   - TTL: 24 часа (как в текущей системе)
   - Одноразовые: флаг `used=FALSE` до использования

### Безопасность

- Токен валидируется на обоих сторонах (бот + бэкенд)
- Коды генерируются криптографически безопасным способом (`secrets.choice()`)
- Токен хранится в переменных окружения (не в коде)

### API контракты

**Новый эндпоинт бэкенда:**

```
POST /api/v1/auth/register-from-invite

Headers:
  X-Bot-Secret: <BOT_SECRET_KEY>
  Content-Type: application/json

Request body:
{
  "tg_id": 123456789,
  "username": "john_doe",
  "first_name": "John",
  "last_name": "Doe",
  "language_code": "en",
  "invite_token": "INVITE_TOKEN"
}

Response (201):
{
  "tg_id": 123456789,
  "login_code": "ABC12345",
  "code_expires_at": "2026-04-28T12:34:56Z"
}

Error responses:
  400 — Invalid invite token
  409 — User already exists (по tg_id)
```

## Архитектура

### Компоненты

| Компонент | Описание |
|-----------|---------|
| **Веб-форма** | Кнопка с ссылкой `t.me/bot?start=INVITE_TOKEN` |
| **Бот (обработчик /start)** | Валидирует токен, проверяет пользователя, генерирует код или вызывает API |
| **Бэкенд (POST /api/v1/auth/register-from-invite)** | Регистрирует новых пользователей, валидирует токен, генерирует код |
| **Таблица login_codes** | Хранит коды с TTL 24 часа (существует, используется как сейчас) |

### Поток данных

```
Случай 1: Пользователь уже в системе
─────────────────────────────────────
Бот получает /start INVITE_TOKEN
  ├─ Валидирует INVITE_TOKEN локально
  ├─ Проверяет: есть ли пользователь с этим tg_id в БД (SELECT FROM users WHERE tg_id=...)
  ├─ Генерирует код (8 символов A-Z+0-9)
  ├─ Сохраняет в login_codes (INSERT, TTL 24ч)
  └─ Отправляет пользователю: "Ваш код для входа: ABC12345"

Случай 2: Новый пользователь
──────────────────────────────
Бот получает /start INVITE_TOKEN
  ├─ Валидирует INVITE_TOKEN локально
  ├─ Проверяет: нет пользователя (SELECT FROM users WHERE tg_id=...)
  ├─ Вызывает POST /api/v1/auth/register-from-invite
  │   └─ Передает: tg_id, username, first_name, last_name, language_code, invite_token
  ├─ Бэкенд:
  │   ├─ Валидирует invite_token
  │   ├─ Создает пользователя в БД (INSERT INTO users)
  │   ├─ Генерирует код (8 символов)
  │   ├─ Сохраняет в login_codes (INSERT, TTL 24ч)
  │   └─ Возвращает код
  ├─ Бот получает код от бэкенда
  └─ Отправляет пользователю: "Ваш код для входа: ABC12345"
```

## Реализация

### Файлы для изменения

**Бэкенд:**
- `backend/app/api/v1/auth.py` — добавить эндпоинт `register_from_invite()`
- `backend/app/services/auth.py` — добавить логику регистрации с валидацией токена
- `backend/app/schemas/auth.py` — добавить схему `RegisterFromInviteRequest` и `RegisterFromInviteResponse`
- `backend/app/core/config.py` — добавить переменную `INVITE_TOKEN`

**Бот:**
- `bot/handlers/commands.py` или новый файл `bot/handlers/start.py` — обработчик `/start` с параметром
- `bot/core/config.py` или `bot/.env` — добавить `INVITE_TOKEN`
- `bot/services/auth.py` или `bot/core/auth.py` — функция для генерации кода и проверки пользователя

**Миграции (опционально):**
- Если нужна новая колонка в `users` для отслеживания источника регистрации (web, bot, etc.)

### Конфигурация

Добавить в `.env` файлы:

```env
# backend/.env
INVITE_TOKEN=YOUR_SECRET_TOKEN_HERE

# bot/.env
INVITE_TOKEN=YOUR_SECRET_TOKEN_HERE
```

## Тестирование

### Unit тесты

1. **Генерация кода**
   - Проверка формата (8 символов, A-Z+0-9)
   - Криптографическая безопасность

2. **Валидация токена**
   - Валидный токен проходит
   - Невалидный токен отклоняется
   - Пустой токен отклоняется

3. **Регистрация пользователя**
   - Новый пользователь создается корректно
   - Дублирующийся tg_id возвращает ошибку 409
   - Код генерируется и возвращается в ответе

### Интеграционные тесты

1. **Существующий пользователь**
   - Бот получает `/start INVITE_TOKEN`
   - Код генерируется и сохраняется
   - Код работает для входа на сайт

2. **Новый пользователь**
   - Бот получает `/start INVITE_TOKEN`
   - Вызывается API регистрации
   - Пользователь создается в БД
   - Код генерируется и сохраняется
   - Код работает для входа на сайт

3. **Ошибки**
   - Неправильный токен → бот отклоняет
   - Бэкенд недоступен → бот выдает ошибку
   - Невалидные данные пользователя → API возвращает 400

### E2E сценарии

1. Открыть веб-форму → клик на кнопку → переход в бота → получение кода → вход на сайт
2. Новый пользователь → все то же + регистрация в системе

## Миграция данных

Не требуется — используются существующие таблицы и структуры.

## Откат

Если что-то пойдет не так:
1. Удалить эндпоинт `register-from-invite` из бэкенда
2. Удалить обработчик `/start` из бота
3. Удалить `INVITE_TOKEN` из конфигурации
4. Вернуться к старому потоку регистрации (только через бота напрямую)

## Примечания

- INVITE_TOKEN может быть простой строкой, например `web_invite_2026`
- Не требуется генерация нового токена для каждого пользователя — один для всех
- Коды-пароли используют существующую таблицу `login_codes`, изменений в БД не требуется
- Логика на боте для проверки существующего пользователя может быть простой: попытка вызвать API бэкенда или локальная проверка в БД бота (если бот имеет доступ к БД)
