# Модуль `getters`

Модуль `getters` в боте `Bot_3xui_vpn` отвечает за получение и подготовку данных, которые отображаются в интерфейсе пользователя.
Он является ключевым компонентом архитектуры, построенной на библиотеке `aiogram-dialog`, и обеспечивает разделение
логики представления (view) и бизнес-логики.

## Назначение

Основная задача модуля `getters`:

- **Формирование данных для окон диалога.** Каждое окно в `aiogram-dialog` требует функции-геттера, которая возвращает словарь
  данных для рендеринга шаблона (Jinja2).
- **Декоративная логика.** Подготовка данных к отображению: форматирование дат, расчет прогресс-баров, генерация
  сообщений с эмодзи и разметкой.
- **Централизованное управление.** Все логика, связанная с отображением, сосредоточена в одном месте, что упрощает
  поддержку и рефакторинг.

## Ключевые компоненты

### 1. `registry.py`

Этот файл является ядром системы управления геттерами.

**Основные классы и функции:**

- `Getter`: Тип-алиас для асинхронной функции, которая принимает `DialogManager` и возвращает `Dict[str, Any]`.
- `GETTER_REGISTRY: Dict[str, Getter]`: Глобальный словарь, который хранит соответствие между именем состояния FSM (например, `MainMenu:main`)
  и соответствующей функцией-геттером.
- `@register_getter(state: str)`: Декоратор, который регистрирует функцию-геттер в `GETTER_REGISTRY` под указанным состоянием.
- `get_getter(state: str) -> Optional[Getter]`: Функция для получения зарегистрированного геттера по имени состояния.

**Принцип работы:**

1.  Разработчик создает асинхронную функцию-геттер, которая получает данные из базы данных или кэша.
2.  Эта функция обрамляется декоратором `@register_getter("ИмяСостояния")`.
3.  Во время запуска бота все модули `getters` импортируются, и декораторы выполняются, регистрируя функции в реестре.
4.  Когда диалог переходит в состояние `ИмяСостояния`, `aiogram-dialog` использует `get_getter` для поиска и вызова
    соответствующей функции, чтобы получить данные для рендеринга окна.

### 2. `base.py`

Файл предоставляет абстрактный базовый класс для создания сервисов, упрощающих формирование окон.

**Класс `BaseService(ABC)`:**

Этот абстрактный класс задает общий интерфейс для сервисов, которые отвечают за конкретное окно или группу окон.

- `__init__(target_state: str)`: Принимает полное имя состояния (например, `MainMenu:main`), для которого сервис будет работать.
- `@abstractmethod async def designer_getters(...) -> Dict[str, Any]`: Абстрактный метод для получения данных (аналог геттера).
- `@abstractmethod def designer_msg() -> Text`: Абстрактный метод для возвращения объекта текста для окна.
- `@abstractmethod def designer_keyboard() -> Keyboard`: Абстрактный метод для возвращения объекта клавиатуры для окна.

**Назначение:** Позволяет объединить логику данных, сообщения и клавиатуры в одном классе, что улучшает организацию кода.

### 3. Организация по подпапкам

Модуль структурирован по функциональным признакам:

- `keys/`: Геттеры, связанные с управлением ключами VPN (просмотр, создание, продление, удаление).
  - `key_management.py`: Основной геттер `display_key_details` для показа деталей ключа и `process_getter_view_keys` для списка ключей.
  - `trial_key.py`: Геттеры для работы с пробными ключами.
- `main/`: Геттеры для главного меню (`main.py`). Класс `MainMenuGetter` реализует `BaseService`.
- `on_click/`: Содержит **обработчики нажатий (handlers)**, а не геттеры. Это, возможно, не совсем корректное размещение.
  - `key_click.py`: Функции для обработки действий с ключами (просмотр, удаление, продление).
- `registor/`: Содержит систему регистрации сервисов (`registrate.py`). Класс `RegistrateService` сопоставляет состояния FSM
  с экземплярами `BaseService`.
- `castoms/`: Кастомные виджеты, например, `CustomCalendar` с модифицированным отображением.

## Важные наблюдения

1.  **Смешение ответственностей:** Подпапка `on_click` содержит обработчики нажатий (`handlers`), которые по своей сути
    не являются геттерами. Это может быть поводом для рефакторинга и перемещения этих функций в папку `handlers`.
2.  **Потенциальные артефакты:** Файлы `admin_getter.py`, `gift_getter.py`, `instruction.py`, `service.py`, `user_utilit.py`,
    `workers.py` содержат закомментированный код или полностью пусты. Это указывает на то, что часть функционала
    находится в процессе разработки или была удалена, но файлы не были очищены.
3.  **Регистрация геттеров:** Поиск показал, что декоратор `@register_getter` используется только в `registry.py`,
    что означает, что реальная регистрация функций в реестре, скорее всего, происходит в другом месте кодовой базы,
    либо система регистрации пока не полностью реализована.

## Заключение

Модуль `getters` представляет собой хорошо структурированную, но, возможно, незавершенную систему для управления
данными интерфейса. Он использует паттерны реестра и базового сервиса для обеспечения гибкости и поддержки.
Основная рабочая логика сосредоточена в `registry.py`, `base.py` и конкретных геттерах в подпапках `keys/` и `main/`.
Для дальнейшего развития проекта рекомендуется завершить реализацию системы регистрации и провести рефакторинг
смешанных папок, таких как `on_click`.
