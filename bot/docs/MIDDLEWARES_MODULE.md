# Модуль `middlewares`

> ⚠️ **ВАЖНО:** После рефакторинга (30 мая 2026) бот стал чистым UI-слоем. `DatabaseMiddleware` и `XUIMiddleware` **удалены** — бот больше не работает с БД и 3x-UI панелью напрямую. Все данные приходят через `BackendAPIClient`.

Модуль `middlewares` в боте `Bot_3xui_vpn` реализует паттерн "Промежуточное ПО" (Middleware),
который позволяет перехватывать и обрабатывать входящие события (сообщения, колбэки) до их передачи в обработчики (handlers).
Это ключевой компонент для централизованной обработки таких задач, как внедрение зависимостей, логирование и обработка ошибок.

## Назначение

Основные цели модуля `middlewares`:

- **Внедрение зависимостей (Dependency Injection):** Централизованное предоставление сервисов (таких как база данных, кэш, API-клиенты)
  всем обработчикам и другим компонентам приложения.
- **Централизованное логирование:** Отслеживание всех событий, входящих в бота, и времени выполнения обработчиков.
- **Обработка ошибок:** Перехват и безопасное реагирование на исключения, возникающие в диалогах.
- **Автоматизация рутинных задач:** Таких как регистрация новых пользователей при их первом заходе.

## Ключевые компоненты

### 1. `dependency_injection.py` — Основа всей системы DI

Этот middleware является фундаментом архитектуры внедрения зависимостей. Он отвечает за инициализацию и предоставление
DI-контейнера (на основе библиотеки `punq`) всем остальным компонентам.

- **`DependencyInjectionMiddleware`**: Главный middleware, который создает один экземпляр контейнера (`Container`) при первом запуске
  (используя `static`-переменную) и добавляет его в словарь `data` события.
- **`get_container()`**: Асинхронная функция в `services/conteiner/app.py`, которая вызывается для инициализации контейнера.
  Поиск показал, что реализация `create_container` (которая должна регистрировать сервисы) отсутствует, что указывает на незавершенность
  системы DI.

### 2. `decorators.py` — Инъекция зависимостей в функции

Файл `decorators.py` содержит декоратор `@inject`, который позволяет автоматически внедрять зависимости в любую асинхронную функцию
на основе аннотаций типов параметров.

- Работает в тесной связке с `DependencyInjectionMiddleware`, используя `container` из `data`.
- Позволяет писать обработчики и сервисы в стиле `async def handler(cache: CacheService, db: DatabaseService)` без явной передачи этих объектов.

### 3. `cache_middleware.py` — Специализированный middleware для кэша

- **`CacheMiddleware`**: Извлекает `CacheService` из контейнера и помещает его в `data["cache"]`. Этот сервис используется
  для быстрого доступа к данным пользователя, ключам и тарифам, полученным из backend API.

**❌ Удалены:**
- `database_mw.py` (`DatabaseMiddleware`) — бот больше не обращается к БД напрямую.
- `xui_middleware.py` (`XUIMiddleware`) — бот больше не взаимодействует с панелью 3x-ui напрямую.

### 4. `registration_users.py` — Автоматическая регистрация пользователей

Этот middleware реализует логику автоматической регистрации пользователей при их первом визите или по реферальной ссылке.

- **Сначала проверяет backend API** (`BackendAPIClient.get_user(user_id)`). Если пользователь найден в backend,
  считает его уже зарегистрированным и пропускает локальную регистрацию.
- Если пользователя нет в backend — перехватывает команду `/start` и извлекает параметр (токен)
  (например, `/start gift_ABC123`).
- Резолвит `RegistrationFactory` и `GiftRegistration` из DI-контейнера как синглтоны.
- Использует `RegistrationFactory` для определения типа регистрации (обычная, по подарку) и обрабатывает ее.
- Результат регистрации (`success`, `type`) сохраняется в `data["registration_result"]`, что позволяет обработчикам
  `/start` в `handlers/start.py` перенаправить пользователя в нужное окно диалога.
- Регистрация компонентов происходит через `services/conteiner/registrate/core/registration.py:RegistrationRegistrar`.

### 5. `dialog_error_handler.py` — Обработка ошибок диалогов

Этот middleware перехватывает специфические ошибки, возникающие в библиотеке `aiogram-dialog`.

- **Ключевая ошибка:** `UnknownIntent`. Эта ошибка возникает, когда пользователь пытается взаимодействовать с диалогом,
  который уже был закрыт (например, нажимает старую кнопку из сообщения в чате). Это частая проблема в Telegram-ботах.
- Если ошибка перехватывается, middleware записывает предупреждение в лог, пытается отправить пользователю случайное
  сообщение об ошибке из списка `error_messages` и спокойно завершает обработку, предотвращая аварийное завершение бота.

### 6. `logging_middlewaries.py` — Централизованное логирование

Этот middleware обеспечивает единообразное и информативное логирование всех событий.

- **Входящие события:** Записывает в лог информацию о входящих сообщениях (ID пользователя, текст) и колбэках (данные колбэка).
- **Выполнение обработчиков:** Фиксирует время начала и завершения выполнения каждого хендлера.
- **Статус выполнения:** Использует разные уровни логирования:
  - `logger.info` для входящих событий (иконка 📥).
  - `logger.success` для успешного завершения (иконка ✅).
  - `logger.error` для возникших исключений (иконка ❌), включая тип и сообщение об ошибке.

## Оптимизации (Этап Оптимизации Middleware через DI)

### RegistrationUsersMiddleware: DI-регистрация компонентов + backend-first check

**Изменения после рефакторинга:**
- Middleware теперь **сначала проверяет backend API** (`BackendAPIClient.get_user()`) вместо локального кэша/БД.
- `GiftRegistration` и `RegistrationFactory` резолвятся из DI-контейнера как синглтоны.

**Решение:** Компоненты регистрируются в DI-контейнере через `RegistrationRegistrar`:
```python
# Регистрирует:
- GiftRegistration (с зависимостью от ServiceDataModel)
- RegistrationFactory (синглтон без зависимостей)
```

**Плюсы:**
- Следует DI-паттерну проекта
- Упрощает тестирование через mock'ирование
- Обеспечивает единый источник правды (backend API)

## Архитектурные наблюдения

1.  **Иерархия middleware:** Существует четкая иерархия. `DependencyInjectionMiddleware` должен выполняться первым,
    чтобы предоставить `container`. Затем `CacheMiddleware` использует этот контейнер для получения кэш-сервиса.
    `RegistrationUsersMiddleware` и `DialogExceptionHandlerMiddleware` используют результаты предыдущих middleware.
    **Удалены:** `DatabaseMiddleware`, `XUIMiddleware` — бот больше не работает с БД и 3x-UI напрямую.
2.  **DI контейнер:** Проект использует легковесный DI-контейнер `punq`. Регистрация сервисов происходит через
    registrar-паттерн в `services/conteiner/registrate/`.
3.  **Разделение ответственностей:** Модуль демонстрирует хорошее разделение ответственностей. Каждый middleware решает
    одну конкретную задачу, что делает код легко тестируемым и поддерживаемым.
4.  **Гибкость:** Использование DI и паттерна middleware делает архитектуру очень гибкой. Новые сервисы можно легко добавить,
    зарегистрировав их в контейнере и внедрив через соответствующий middleware или resolver.
5.  **Оптимальное разделение ролей:** Middleware используются для инъекции контекста события (кэш, API-клиенты),
    а DI контейнер — для синглтонов и сложных сервисов с множеством зависимостей.

## Заключение

Модуль `middlewares` является сердцем архитектуры бота `Bot_3xui_vpn`. После рефакторинга он сфокусирован на
UI-слое: DI, кэш, регистрация через backend API, логирование и обработка ошибок. Бот больше не управляет соединениями с БД
и панелью 3x-ui напрямую — эта ответственность полностью делегирована backend'у.
