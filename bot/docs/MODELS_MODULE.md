# Модуль `models`

Модуль `models` в боте `Bot_3xui_vpn` представляет собой централизованное хранилище для **объектов данных** (Data Transfer Objects - DTO),
которые используются для передачи информации между различными компонентами системы: обработчиками (handlers), сервисами (services),
базой данных и кэшем. Он основан на паттерне `dataclass` из стандартной библиотеки Python.

## Назначение

Основные цели модуля `models`:

- **Стандартизация данных:** Предоставление четко определенных и типизированных структур для всех основных сущностей приложения.
- **Изоляция бизнес-логики от хранения:** Объекты моделей не содержат логики взаимодействия с базой данных (это делают сервисы) или кэшем,
  они просто описывают, как данные должны выглядеть.
- **Упрощение сериализации/десериализации:** Методы `to_dict()` и `from_dict()` позволяют легко конвертировать объекты в словари
  (для хранения в JSON или передачи в шаблоны) и обратно.
- **Централизованное управление:** Все определения данных находятся в одном месте, что упрощает поддержку и рефакторинг.

## Ключевые компоненты (Основные модели)

### 1. `models/users/user.py` — `User`

Описывает основные данные о пользователе Telegram.

- **Поля:** `tg_id`, `username`, `first_name`, `language_code`, `is_admin`, `trial` (флаг пробного периода), `server_id` (ссылка на сервер),
  `referral_id` (ID пригласившего пользователя), `is_blocked`.
- **Особенности:** Использует `__post_init__` для автоматической установки таймстампов `created_at` и `updated_at` при создании объекта.

### 2. `models/keys/key.py` — `Key`

Описывает VPN-ключ пользователя.

- **Поля:** `tg_id` (владелец), `client_id`, `email`, `expiry_time` (таймстамп в мс), `key` (публичная ссылка), `inbound_id` (ссылка на сервер),
  `total_gb` (общий лимит трафика), `used_traffic` (использованный трафик), `tariff_id`, `amount` (стоимость), `limit_ip`, `period`.
- **Особенности:** Содержит свойство `warp_expiry_time`, которое возвращает форматированную дату истечения.

### 3. `models/servers/server.py` и `models/servers/inbound.py` — `Server`, `Inbound`

Описывают инфраструктуру VPN.

- **`Server`:** `id`, `cluster_name`, `server_name`, `api_url` (для управления через 3x-ui), `subscription_url` (для подключения клиента), `login`, `password`.
- **`Inbound`:** `server_id`, `inbound_id`, `name_inbound`. Обозначает конкретный "вход" (inbound) на сервере, к которому можно привязать ключ.

### 4. `models/tariffs/tariff.py` — `Tariff`

Описывает тарифный план.

- **Поля:** `id`, `name_tariff`, `amount` (цена в рублях), `description`, `limit_ip`, `period` (длительность в днях), `traffic_limit` (лимит трафика в ГБ).

### 5. `models/payments/payment.py` — `PaymentModel`

Описывает информацию о платеже.

- **Поля:** `payment_id` (ID от платежной системы), `tg_id`, `amount`, `payment_type` (например, "renew_key|email"), `status`, `created_at`.

### 6. `models/referrals/` — Реферальная система

Группа моделей, описывающих партнерскую программу.

- **`Referral`:** Основной объект, связывающий реферала (`referral_id`) и пригласившего (`referrer_id`), с полем `discount_percent`.
- **`ReferralLink`:** Реализация реферальной ссылки. Содержит `token` и `referrer_tg_id`.
- **`ReferralRedemption`:** Фиксирует факт использования реферальной ссылки (`referred_tg_id`).
- **`ReferralReward`:** Описывает награду, выданную пригласившему (например, скидку или бонус).

### 7. `models/gifts/gift_link.py` — `GiftLink`

Описывает подарочный ключ.

- **Поля:** `sender_tg_id`, `tariff_id`, `token`, `recipient_tg_id`, `used_at`, `_status`.
- **Особенности:** Является наиболее сложной моделью, так как содержит **бизнес-логику** в виде методов `is_redeemable()`, `redeem()` и `is_expired()`.
  Это отличает ее от других моделей, которые являются чистыми DTO. Этот подход позволяет инкапсулировать правила поведения подарка в самом объекте.

## Вспомогательные модели

- **`models/cache.py`:** Содержит вспомогательные классы для кэша.
  - `CacheItem`: Обертка для любого значения, которое нужно хранить в кэше с возможностью установки срока действия (`expires_at`).
  - `REGISTRATE_USER`: DTO для временных данных о регистрации пользователя, используемый в middleware.
- **`models/mass_mailing.py` — `MassMailing`:** DTO для рассылки сообщений, содержит `id`, `title`, `emoji`.

## Архитектурные наблюдения

1.  **Использование `dataclass`:** Это правильный и современный выбор. Он автоматически генерирует методы `__init__`, `__repr__` и `__eq__`,
    что значительно сокращает объем шаблонного кода.
2.  **Принцип `to_dict` / `from_dict`:** Унифицированный интерфейс для сериализации упрощает интеграцию с системами хранения данных,
    такими как Redis (кэш) и база данных.
3.  **Глубокая вложенность директорий:** Структура `models/сущность/__init__.py` и `models/сущность/сущность.py` может показаться
    избыточной для простых DTO, но она обеспечивает четкое разделение и масштабируемость. Каждое поддерево может содержать
    дополнительные файлы (например, валидаторы) в будущем.
4.  **Импорты в `__init__.py`:** Главный `__init__.py` в `models` экспортирует все модели, что позволяет импортировать их из единой точки
    (например, `from models import User, Key`), делая код чище.
5.  **Широкое использование:** Поиск в файлах `services` и `getters` показал, что модели импортируются почти везде,
    что подтверждает их центральную роль в архитектуре. Сервисы возвращают объекты моделей, а геттеры используют их для рендеринга.

## Заключение

Модуль `models` является фундаментальным слоем приложения `Bot_3xui_vpn`. Он обеспечивает **типобезопасность** и **предсказуемость**
в обмене данными между компонентами. Хорошо продуманная структура и использование современных возможностей Python (dataclass)
делают этот модуль удобным в использовании и поддержке. Наличие бизнес-логики в `GiftLink` — это интересное отклонение
от чистого DTO-паттерна, которое оправдано сложностью предметной области подарков. В целом, модуль демонстрирует высокое качество
архитектурного решения.
