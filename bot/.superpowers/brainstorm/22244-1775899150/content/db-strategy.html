Стратегия работы с базой данных

Веб-бэкенд и Telegram-бот будут сосуществовать — как они делят данные?

A

Общая БД (прямой доступ)

Бот и веб-бэкенд работают с одной PostgreSQL. Веб-бэкенд использует те же репозитории.

Плюсы

  • Просто — нет дублирования логики
  • Данные всегда актуальны
  • Быстрый старт

Минусы

  • Нужна осторожность при миграциях
  • Конкуренция за соединения
B

Веб-бэкенд вызывает API бота

Бот — источник истины. Веб-бэкенд делает HTTP-запросы к внутреннему API бота для любых изменений.

Плюсы

  • Вся бизнес-логика в одном месте
  • Бот контролирует все операции

Минусы

  • Нужно писать внутренний API в боте
  • Больше сетевых вызовов
  • Сложнее в отладке
C

Отдельная БД + синхронизация

Веб-бэкенд имеет свою схему. Данные синхронизируются через события или периодически.

Плюсы

  • Полная независимость
  • Можно оптимизировать схему под web

Минусы

  • Сложность синхронизации
  • Риск рассинхронизации данных
  • Избыточно для MVP
