# Замечания по фронтенду VPN Web Backend

> Выявлены в процессе написания и запуска E2E тестов (Playwright)
> Дата: 2025-04-12

---

## 1. Аутентификация и авторизация

### 1.1. Одинаковое сообщение об ошибке для любых проблем логина
**Серьёзность:** 🔴 Высокая

Фронтенд показывает одно и то же сообщение «Необходима авторизация» при:
- Неверном пароле
- Несуществующем email
- Пустых полях
- Любых других ошибках аутентификации

**Ожидаемое поведение:** Разные сообщения для разных ошибок:
- «Неверный email или пароль» — при неправильных данных
- «Пользователь не найден» — при несуществующем email
- «Заполните все обязательные поля» — при пустых полях

**Где искать:** `frontend/index.html` — обработка ответа от `/api/v1/auth/login`

---

### 1.2. Асинхронная обработка формы логина
**Серьёзность:** 🟡 Средняя

После нажатия кнопки «Войти» редирект на `#/dashboard` происходит не мгновенно. В некоторых случаях требуется ожидание 2-4 секунды. При автоматическом тестировании это вызывает race conditions.

**Рекомендация:** Добавить визуальный индикатор загрузки (спиннер на кнопке) и убедиться что `window.location.hash` обновляется сразу после успешного ответа API.

---

### 1.3. Отсутствие валидации формы на стороне клиента
**Серьёзность:** 🟡 Средняя

Форма логина отправляет запрос на сервер даже с пустыми полями. Нет клиентской валидации `required` или `type="email"`.

**Ожидаемое поведение:** HTML5 атрибуты `required` и `type="email"` должны блокировать отправку формы с некорректными данными.

---

## 2. Регистрация

### 2.1. Нестабильные селекторы элементов
**Серьёзность:** 🟡 Средняя

Инпуты на странице регистрации имеют ID `#regEmail`, `#regPassword`, `#regTgId`, но при этом:
- Страница может не отрендериться полностью при быстром переходе
- Нет визуального индикатора загрузки страницы

**Рекомендация:** Добавить `data-testid` атрибуты для всех ключевых элементов формы.

---

### 2.2. Неясное поведение после регистрации
**Серьёзность:** 🟡 Средняя

После успешной регистрации редирект может происходить на `#/dashboard` или оставаться на `#/register` в зависимости от состояния. Нет чёткого toast-уведомления об успехе.

---

## 3. Навигация и роутинг

### 3.1. Кнопка logout не найдена по селектору `.btn-logout`
**Серьёзность:** 🔴 Высокая

В header отсутствует элемент с классом `.btn-logout` или он имеет другой селектор. Это делает невозможным автоматический выход из системы.

**Рекомендация:** 
- Добавить стабильный селектор (например `data-testid="logout-btn"`)
- Или актуализировать класс кнопки выхода

---

### 3.2. Ссылка на admin видна обычному пользователю
**Серьёзность:** 🟡 Средняя

Тест `test_admin_link_hidden_for_regular_user` проходит, но есть подозрение что ссылка на админ-панель может отображаться в навигации для всех пользователей (скрывается только CSS).

**Ожидаемое поведение:** Ссылка на `#/admin` не должна рендериться в DOM для обычных пользователей.

---

## 4. Dashboard (VPN ключи)

### 4.1. JavaScript ошибка при загрузке dashboard
**Серьёзность:** 🟡 Средняя

В консоли браузера фиксируется ошибка:
```
TypeError: this._periodStr is not a function
    at dashboard (http://localhost:8000/:1403:43)
```

Несмотря на ошибку, страница загружается и ключи отображаются. Однако это указывает на потенциальный баг в логике отображения периодов.

**Где искать:** `frontend/index.html`, функция `dashboard`, строка ~1403

---

### 4.2. Отсутствие спиннера загрузки на dashboard
**Серьёзность:** 🟢 Низкая

Элемент `.loading-spinner` не появляется при загрузке страницы, из-за чего тесты не могут использовать его как индикатор готовности страницы.

**Рекомендация:** Добавить спиннер на время fetch запроса ключей.

---

## 5. Селекторы и тестируемость

### 5.1. Отсутствие `data-testid` атрибутов
**Серьёзность:** 🟡 Средняя

Все элементы идентифицируются по CSS классам или ID которые могут измениться при рефакторинге.

**Рекомендация:** Добавить `data-testid` для всех ключевых элементов:

| Элемент | Рекомендуемый data-testid |
|---|---|
| Email input (login) | `login-email` |
| Password input (login) | `login-password` |
| Login button | `login-submit` |
| Telegram login button | `login-telegram` |
| Register link | `link-to-register` |
| Email input (register) | `register-email` |
| Password input (register) | `register-password` |
| TG ID input (register) | `register-tg-id` |
| Register button | `register-submit` |
| Logout button | `logout-button` |
| Key cards grid | `keys-grid` |
| Create key button | `create-key-button` |
| Key renew button | `key-renew-{index}` |
| Key delete button | `key-delete-{index}` |
| Key copy button | `key-copy-{index}` |
| Admin users tab | `admin-tab-users` |
| Admin keys tab | `admin-tab-keys` |
| Toast container | `toast-container` |
| Modal overlay | `modal-overlay` |

---

### 5.2. Несогласованность именование ID
**Серьёзность:** 🟢 Низкая

На странице логина: `#loginEmail`, `#loginPassword`
На странице регистрации: `#regEmail`, `#regPassword` (сокращение `reg` вместо `register`)

**Рекомендация:** Единый стиль именов: `#login-email`, `#login-password`, `#register-email`, `#register-password`.

---

## 6. Toast уведомления

### 6.1. Неточные тексты ошибок
**Серьёзность:** 🟡 Средняя

Текст «Необходима авторизация» появляется при ошибке входа, что вводит пользователя в заблуждение.

**Рекомендация:** Сопоставить тексты тостов с реальными ошибками API:

| Ошибка API | Текущий текст | Ожидаемый текст |
|---|---|---|
| 401 Unauthorized | «Необходима авторизация» | «Неверный email или пароль» |
| 404 User not found | «Необходима авторизация» | «Пользователь не найден» |
| 422 Validation error | (неизвестно) | «Заполните все обязательные поля» |
| 500 Server error | (неизвестно) | «Ошибка сервера, попробуйте позже» |

---

### 6.2. Toast содержит символ эмодзи ✕
**Серьёзность:** 🟢 Низкая

Текст тоста начинается с символа `✕`: `✕Необходима авторизация`. Это может быть иконкой закрытия которая сливается с текстом.

**Рекомендация:** Отделить иконку закрытия от текста (например через CSS `::before` или отдельный элемент).

---

## 9. Доступность (a11y)

### 9.1. Отсутствуют label у инпутов
**Серьёзность:** 🟡 Средняя

Инпуты используют только `placeholder` для обозначения назначения. Отсутствуют `<label>` элементы или `aria-label`.

**Рекомендация:** Добавить `<label for="...">` или `aria-label` ко всем инпутам.

---

**Серьёзность:** 🟢 Информационная

Навигация Tab работает корректно на странице логина. Фокус виден на инпутах.
