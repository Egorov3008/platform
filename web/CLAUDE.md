## Architecture Overview (Post-Stage 3)

This is a **stateless FastAPI API proxy + auth layer** for a VPN web interface. All business operations are delegated to the backend API via HTTP.

**Request Flow:**
```
Web Frontend (SPA)
    ‚Üì
Web API Layer (app/api/*)
    ‚îú‚îÄ Auth endpoints: JWT cookies, login_codes (local DB)
    ‚îî‚îÄ Business endpoints: HTTP proxy to Backend API
    ‚Üì
httpx.AsyncClient (WebBackendClient)
    ‚Üì
Backend API (/api/v1/keys, /payments, /tariffs, /users)
```

### Database (Auth Only)

The web layer maintains **only authentication tables** in PostgreSQL:
- `login_codes` ‚Äî 8-char alphanumeric codes generated by `/api/v1/auth/generate-code`, 24h TTL, single-use
- `web_users` ‚Äî web session tracking, linked to Telegram `tg_id`
- `magic_tokens` ‚Äî magic link auth tokens (if used)

**All business data is accessed via backend API** (keys, payments, tariffs, users, servers). The web layer has **zero direct access** to VPN keys, payments, or user data tables.

### Authentication

JWT-based (python-jose) with unified Telegram Widget support for new and existing users:
- **Access + Refresh tokens** stored in HttpOnly cookies (`access_token`, `refresh_token`)
- **CSRF protection** via non-HttpOnly `csrf_token` cookie + `X-CSRF-Token` header
- **Login flow** (unified for all users):
  ```
  User clicks Telegram Widget on login page
      ‚Üì
  POST /api/v1/auth/telegram-callback
      ‚îú‚îÄ Verify CAPTCHA signature
      ‚îú‚îÄ Verify Telegram Widget data signature
      ‚îú‚îÄ Extract tg_id from Telegram data
      ‚îÇ
      ‚îú‚îÄ GET /users/{tg_id} (to backend) ‚Äî check if user exists
      ‚îÇ   ‚îú‚îÄ If 404 (new user): POST /users {tg_id} ‚Üí auto-create in backend
      ‚îÇ   ‚îî‚îÄ If 200 (existing user): use existing data from backend
      ‚îÇ
      ‚îú‚îÄ Generate JWT with tg_id and is_admin flag
      ‚îú‚îÄ Save session in web_users table
      ‚îî‚îÄ Return cookies + JWT
      ‚Üì
  User logged in (both new and existing paths converge to same result)
  ```

**Key Design Points:**
- **New users** (not in backend): automatically created via `POST /users {tg_id}`
- **Existing users** (from bot): directly use existing data from backend
- **Same entry point** for both: Telegram Widget (no separate flows)
- **Transparent to frontend**: Both paths return identical JWT structure

### WebBackendClient

HTTP client (`app/api/backend_client.py`) handles all business operations:
- **Methods**: `list_tariffs()`, `create_key(tariff_id)`, `delete_key(email)`, `renew_key(email, tariff_id)`, etc.
- **Authentication**: All requests include `X-Bot-Secret: <BOT_SECRET_KEY>` header (service-to-service auth)
- **Dependency**: Injected via `get_backend_client(request, current_user)` ‚Äî extracts user's `tg_id` from JWT
- **Error handling**: httpx exceptions propagate as HTTP responses (400, 402, 404, 500, etc.)

### API Endpoints

**Auth (local):**
- `POST /api/v1/auth/login` ‚Äî verify code, set cookies
- `POST /api/v1/auth/logout` ‚Äî clear cookies
- `POST /api/v1/auth/refresh` ‚Äî refresh JWT via refresh token

**Keys (proxied to backend):**
- `GET /api/v1/keys/` ‚Äî list user's VPN keys
- `POST /api/v1/keys/` ‚Äî create new key (free tariffs)
- `GET /api/v1/keys/{email}` ‚Äî get key details
- `POST /api/v1/keys/{email}/renew` ‚Äî renew key expiry (free only)
- `DELETE /api/v1/keys/{email}` ‚Äî delete key

**Tariffs (proxied to backend):**
- `GET /api/v1/tariffs/` ‚Äî list all tariffs
- `GET /api/v1/tariffs/{id}` ‚Äî get tariff details

**Payments (proxied to backend):**
- `GET /api/v1/payments/` ‚Äî list user's payment history
- `POST /api/v1/payments/create` ‚Äî initiate payment (creates YooKassa invoice)
- `GET /api/v1/payments/{payment_id}/status` ‚Äî check payment status

### Dependencies & Services Removed

**Removed in Stage 3** (business logic moved to backend):
- `app/services/keys.py`, `tariffs.py`, `payments.py` ‚Äî replaced by WebBackendClient
- `app/repositories/keys.py`, `tariffs.py`, `payments.py`, `users.py`, `stocks.py`, `referral.py` ‚Äî data now in backend
- `app/api/bot.py` ‚Äî bot now calls backend directly, web doesn't need bot endpoints
- APScheduler jobs ‚Äî background tasks (cache sync, notifications) now in backend

**Kept** (auth-specific):
- `app/services/auth.py` ‚Äî JWT generation, login code verification
- `app/repositories/login_codes.py`, `web_users.py`, `magic_tokens.py` ‚Äî local auth tables

### Logging

Structured logging via `app/core/logging.py`. Use `get_logger(__name__)` in every module. Configurable via `LOG_LEVEL`, `LOG_FILE`, `LOG_FORMAT` env vars.

## Testing Patterns

Tests mock `WebBackendClient` instead of asyncpg. The `mock_backend` fixture returns an `AsyncMock` object.

```python
@pytest.fixture
def mock_backend():
    return AsyncMock(spec=WebBackendClient)

def test_list_keys(client, mock_backend, mock_current_user):
    app.dependency_overrides[get_backend_client] = lambda: mock_backend
    mock_backend.list_keys.return_value = [{"email": "...", "client_id": "..."}]
    
    response = client.get("/api/v1/keys/")
    assert response.status_code == 200
    
    app.dependency_overrides.clear()
```

Auth tests still mock local repositories (LoginCodesRepo, WebUsersRepo).

## Environment Variables

Required in `.env`:
- `DATABASE_URL` ‚Äî asyncpg DSN for auth tables only (login_codes, web_users, magic_tokens)
- `SECRET_KEY` ‚Äî JWT signing key
- `BOT_SECRET_KEY` ‚Äî shared secret with backend (`X-Bot-Secret` header)
- `BACKEND_URL` ‚Äî backend API base URL (default: `http://localhost:8000`)
- `CSRF_ENABLED` ‚Äî set to `false` to disable CSRF checks (tests/dev only)
- `LOG_LEVEL` ‚Äî logging level (default: INFO)
